🚨 Règle d'or à ne JAMAIS oublier

❌ N'ENVELOPPEZ PAS Vuetify

Ne créez JAMAIS de wrapper pour des composants Vuetify sans raison valable.

Mauvais exemple :

<MyButton />
<MyCard />
<MyInput />

✅ UTILISEZ Vuetify directement

Utilisez VBtn, VCard, VTextField directement. Passez des composables et handlers en props si nécessaire.

Bon exemple :

<VBtn @click="handleClick">
<VCard>
<VTextField v-model="value">

🚨 VUETIFY FIRST - Ne créez PAS de wrappers inutiles

❌ L'erreur la plus commune : Wrapper tous les composants

Beaucoup de développeurs ont le réflexe de créer un wrapper pour "personnaliser" ou "contrôler" les composants. C'est une mauvaise pratique qui viole KISS et crée de la complexité inutile.

❌ MAUVAIS - Créer des wrappers

<!-- ❌ NE FAITES PAS ÇA -->
<script setup>
defineProps({
  label: String,
  color: String,
  onClick: Function
})
</script>

<template>
  <VBtn 
    :color="color"
    @click="onClick"
  >
    {{ label }}
  </VBtn>
</template>

Pourquoi c'est mauvais :

  • ❌ Viole KISS (complexité inutile)
  • ❌ Duplication de l'API Vuetify
  • ❌ Maintenance difficile
  • ❌ Perte des features Vuetify

✅ BON - Utiliser Vuetify directement

<!-- ✅ FAITES ÇA -->
<script setup>
import { useProducts } from '../useProducts'

const { addProduct, isLoading } = useProducts()

const handleAdd = () => {
  addProduct()
}
</script>

<template>
  <VBtn 
    color="primary"
    :loading="isLoading"
    @click="handleAdd"
  >
    Ajouter un produit
  </VBtn>
</template>

Pourquoi c'est bien :

  • ✅ Respecte KISS (simple et direct)
  • ✅ Accès complet à l'API Vuetify
  • ✅ Pas de maintenance supplémentaire
  • ✅ Composable pour la logique

✅ Règles à suivre TOUJOURS

  • Utilisez Vuetify directement
    VBtn, VCard, VTextField, VSelect... tous doivent être utilisés sans wrapper
  • Logique dans composables
    Toute logique métier va dans un composable (useProducts(), useAuth())
  • Handlers dans la page
    Les gestionnaires d'événements sont définis dans la page et passés aux composants
  • Créez un Organism seulement si
    C'est une section complète réutilisée 3+ fois (ProductCard, UserProfile...)
  • Ne créez JAMAIS
    MyButton, MyInput, MyCard, etc. → Utilisez V* directement

Les 5 principes fondateurs

1. DRY - Don't Repeat Yourself

Règle d'or : Chaque concept doit avoir une seule source de vérité dans le code.

✅ Application du DRY

  • Logique dupliquée → Composable
    Si vous copiez-collez de la logique entre 2 pages → créez un composable
  • Fonctions utilitaires → Utils
    formatPrice() utilisé partout → un seul fichier utils/formatters.js
  • État partagé → Store
    Données utilisateur dans 5 pages → un store userStore.js

2. KISS - Keep It Simple, Stupid

Règle d'or : La solution la plus simple est souvent la meilleure.

✅ Application du KISS

  • Composant simple → Vuetify directement
    Besoin d'un bouton ? → <VBtn> sans wrapper
  • Fonction pure → Utils simple
    Pas de classe complexe si une fonction suffit
  • État local → ref() simple
    Pas de store si une seule page utilise les données

❌ Violation du KISS

  • ❌ Créer un wrapper custom pour chaque composant Vuetify
  • ❌ Système de plugins complexe pour 2 lignes de code
  • ❌ Abstractions prématurées "au cas où"

3. DDD - Domain-Driven Design

Règle d'or : Organisez le code par domaine métier, pas par type technique.

📁 src/domains/
  📁 products/
    📄 ProductsPage.vue
    📄 useProducts.js
    📄 productService.js
  📁 orders/
    📄 OrdersPage.vue
    📄 useOrders.js
  📁 auth/
    📄 LoginPage.vue
    📄 useAuth.js

4. Atomic Design

Règle d'or : Construire l'UI du plus petit au plus grand.

  • ⚛️ Atoms = Vuetify (VBtn, VCard...) - NE PAS WRAPPER
  • 🦠 Organisms = Sections complètes (ProductCard.vue)
  • 📄 Pages = Orchestration finale

5. Single Responsibility Principle

Règle d'or : Un fichier = une seule raison de changer.

  • formatters.js → Formatage uniquement
  • validators.js → Validation uniquement
  • productService.js → API produits uniquement
  • useProducts.js → Logique produits uniquement
